## 2026-05-30 — Gemini 2.0 Flash → 2.5 Flash; model now env-overridable #incident #decision

First real Gemini run came back 0/200 parsed in 30 seconds — every call returned `429 RESOURCE_EXHAUSTED` with the diagnostic `limit: 0, model: gemini-2.0-flash`. Three free-tier metrics all showed limit=0, meaning Google pulled `gemini-2.0-flash` from the free tier entirely sometime in 2026. The model name I'd encoded was stale. Fix: switched the default to `gemini-2.5-flash` and made the model env-overridable via `GEMINI_MODEL` so the next time Google reshuffles available free models (which will happen again), no code change is needed. Also added `GEMINI_MIN_INTERVAL_MS` env override since 2.5 Flash's free tier is tighter than 2.0's was (~10 RPM vs. 15 RPM) — default interval bumped from 4.5s to 6.5s. Pricing constants updated to 2.5 Flash's published rates ($0.30/M input, $2.50/M output). Lesson banked: any "default model" string in a vendor-SDK call site should be env-overridable. Hard-coding the model is a footgun every time the vendor reshuffles their tier coverage. Updated CLAUDE.md / README / scope.md / CONTRIBUTING.md / dashboard chip + sidebar + topbar + .env.example to reflect 2.5; also swept stale "Llama" references that survived previous pivots through scope.md sub-features 2-4.

## 2026-05-30 — Pivot: Llama 4 via Groq → Gemini 2.0 Flash via Google AI Studio #pivot #decision

Third LLM commit on Helm. Claude (2026-05-28) was dropped for cost; Llama 4 Scout via Groq (2026-05-28) was the cheap-and-free answer until the Groq free tier's 500K-tokens-per-day cap blocked the full 200-invoice corpus at ~70 invoices on two consecutive days. Today's Groq run: 70 parsed, 130 returned `429 TPD`. Image input on Groq's Llama 4 Scout costs ~6,400 tokens per PNG (per-pixel tile counting), so even a shrunk 1024×1320 image at DSR=1 averaged ~7K tokens per invoice. 200 × 7K = 1.4M tokens needed for one corpus pass, far above the daily ceiling. Switched to Gemini 2.0 Flash via Google AI Studio. Free tier: 15 RPM, 1,500 RPD, 1M TPM, no daily token cap. Image input flat-rates at ~258 tokens per PNG (Gemini uses a tile-based image encoder, not per-pixel patches), so a full 200-invoice run is ~360K total tokens — well under any limit, and at the published paid-tier rate ($0.10/M in, $0.40/M out) it would cost roughly $0.04 if you paid. Wired `@google/genai` (Google's unified Gemini SDK, replaces the legacy `@google/generative-ai`) and built `back/src/ap/extraction-gemini.ts` with rate-limit pacing built in — the extractor tracks the timestamp of the most-recent call and self-paces at 4.5s minimum interval, so the measurement loop doesn't need to know about RPM limits. Gemini's `responseSchema` config constrains output JSON server-side; combined with the existing Zod parser on the client, schema drift becomes effectively impossible (and the retry-on-parse-failure path becomes near-dead-code). Kept `extraction-groq.ts` alongside as an alternative provider — the Extractor interface is provider-agnostic by design and having a second implementation in the repo is a portfolio strength, not maintenance burden. Locked-stack docs (CLAUDE.md, README, scope.md, CONTRIBUTING.md, .env.example) all updated to Gemini-first; Groq footnoted. Dashboard updated: `EXTRACTOR_LABELS` map renders the right model name in the MEASURED · ... badge based on `report.extractor`; chip row shows `model gemini-2.0-flash · provider google ai studio`; sidebar stack panel matches. Still pending: actual Gemini measurement run (user needs to add `GEMINI_API_KEY` to `.env`). Lesson banked: when a "free tier" wall blocks a critical workload, the right move is provider-agnostic interface + parallel implementations, not begging the rate-limiter.

## 2026-05-29 — Groq free-tier daily cap discovered; renderer tightened #incident #decision

First real Groq Llama 4 Scout measurement run surfaced two surprises. (1) The free tier's 30 RPM rate limit was easy to absorb with SDK `maxRetries`. (2) The free tier's **500 K-tokens-per-day cap** was not. Each invoice burned ~6,250 tokens — roughly 825 just for the 2048×2640 px PNG (DSR=2 amplifies image cost more than I appreciated) plus ~1,500 prompt + ~300 output, plus retry attempts that also count toward the budget. The first 200-invoice run cleared 80 invoices (94.5% field accuracy, 88.7% line-item recall — directionally real Groq numbers) before tripping the daily cap. Worse, the followup 20-invoice run was issued from the same daily window and overwrote the 80-invoice `report.json` with twenty quota-exhausted error records before I noticed the measurement script always overwrites the same output path. Lost the 80-invoice data. Lesson: when running real measurements with non-recoverable side effects (cost, quota burn), the script should write to a timestamped subdirectory, not a single canonical output dir. Filed as a parking-lot item; for now, restored the mock baseline from HEAD~. Tightened the PNG renderer to viewport 1024×1320 at DSR=1 (down from DSR=2) — produces ~210 image tokens per invoice instead of ~825 (~4× reduction). A future 200-invoice run should fit in roughly 400 K daily tokens with comfortable headroom.

## 2026-05-29 — Pivot: Supabase → libsql (SQLite + vector) #pivot #decision

User caught Supabase in `.env.example` and questioned whether it was actually in the stack. It wasn't — those env vars were aspirational. Locked libsql in instead via `@libsql/client`. Rationale: data volume is tiny (a few thousand rows total, ten KB articles to embed), libsql ships native vector support (`F32_BLOB` + `vector_distance_cos`) so no separate pgvector setup is needed, and the dev story is a single file at `data/helm.db` with zero account/Docker overhead. Deploy path stays open — Turso Cloud has a free tier and the SQL is identical between local file and remote URL. Hiring-manager-readable story is *better* with libsql than Supabase here: "I picked the right tool for 300 rows of vector data" reads as engineering judgement, not pulling-in-a-managed-service-because-it's-cool. Same provider-agnostic principle as the Llama pivot — adopting libsql doesn't lock out a future Supabase swap if a real-deployment context needs it.
